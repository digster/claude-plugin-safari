# 2026-03-17 — Fix Safari Extension Not Appearing in Preferences

## Problem
The extension built successfully but was invisible in Safari → Settings → Extensions. Root cause: `ENABLE_APP_SANDBOX = NO` on the extension target. Safari requires sandboxed extensions.

## Changes Made

### 1. project.pbxproj
- `ENABLE_APP_SANDBOX = NO` → `YES` for extension target (Debug + Release)
- `MACOSX_DEPLOYMENT_TARGET = 10.14` → `14.0` for extension target (Debug + Release)
- `ENABLE_USER_SELECTED_FILES = readonly` → `readwrite` for containing app (needed for NSSavePanel)

### 2. SafariWebExtensionHandler.swift — Full rewrite
- Replaced `Process()` with `NSUserUnixTask` for sandbox-compatible CLI execution
- Helper script (`run-claude.sh`) loaded from `~/Library/Application Scripts/<extension-bundle-id>/`
- Stdout/stderr captured via `Pipe` file handles
- Removed `#available` guards (macOS 14.0+ only now)
- `verifyCli` now checks helper script presence instead of CLI binary path

### 3. ViewController.swift — Setup flow added
- Added `installHelperScript()` method using `NSSavePanel`
- Panel pre-configured to extension's Application Scripts directory
- Writes `run-claude.sh` and sets executable permission (chmod 755)
- Checks script presence on load and updates UI via `showSetupStatus()`

### 4. Main.html — Setup UI
- Added setup section with "Install Helper Script" / "Reinstall" buttons
- Green status indicator when script is installed

### 5. Script.js — Setup handlers
- Added `showSetupStatus(installed)` function
- Added click handlers for install/reinstall buttons

### 6. Style.css — Setup styles
- Divider, setup section layout, `.setup-ok` green color, `.secondary` button style

### 7. ARCHITECTURE.md — Updated
- Reflected NSUserUnixTask approach, helper script, NSSavePanel installation
- Updated data flow and build instructions

## Verification Steps
1. Clean build (Shift+Cmd+K then Cmd+R)
2. Run containing app → click "Install Helper Script" → save to pre-selected path
3. Safari → Settings → Extensions → "Claude Assistant" should appear
4. Toggle ON, test from toolbar popup

# 2026-03-17 (session 2) — Pre-authorize CLI Tools via `--allowedTools`

## Problem
CLI runs headlessly via NSUserUnixTask (no TTY). When Claude needs interactive permission to use tools like `WebFetch`, it can't prompt the user — responds with "please approve in your terminal" but there's no terminal.

## Changes Made

### 1. ViewController.swift — Helper script v2 + version detection
- Updated script template: `shift 3` + `"$@"` to forward extra CLI flags beyond the base 3
- Added `scriptVersionMarker` constant to detect outdated scripts
- `webView(didFinish:)` reads installed script, checks for `"shift 3"`, passes `upToDate` to JS
- Install success callback now passes `showSetupStatus(true, true)`

### 2. SafariWebExtensionHandler.swift — Parse and forward allowedTools
- Parses `allowedTools` array from native message
- Builds extended arguments list: `[cliPath, prompt, format, --allowedTools, tool1, --allowedTools, tool2, ...]`
- Detects outdated scripts when allowedTools are requested (checks for "shift 3" in script content)
- Returns `setupRequired: true` error if script is outdated

### 3. background.js — allowedTools in settings and native message
- Added `allowedTools: 'WebFetch,WebSearch'` to `DEFAULT_SETTINGS`
- In `runClaude()`: parses comma-separated string to array, includes in native message payload

### 4. settings.html + settings.js — Allowed Tools UI
- Added "Allowed Tools" text input between CLI Path and Save button
- Hint text explains usage and common tools
- Wired up load/save for `allowedTools` setting

### 5. Script.js + Main.html — Version detection UI
- `showSetupStatus(installed, upToDate)` now takes two params
- Shows "update-needed" notice with reinstall button when script is outdated
- Added `#update-needed` div to Main.html with explanatory message

### 6. tests/background.test.js — 8 new tests
- DEFAULT_SETTINGS.allowedTools exists and contains expected tools
- Comma-separated parsing with whitespace trimming
- Empty string → empty array
- allowedTools array present in native message payload

### 7. Documentation
- ARCHITECTURE.md: updated storage schema, data flow, design decisions
- README.md: added allowed tools configuration note, fixed sandbox description

# 2026-03-17 (session 3) — Fix URL-Specific Cache in Popup

## Problem
Popup displayed cached `lastResult` from a previous URL when the user navigated to a different page. The `lastResult` object already stored a `url` field, but the popup never validated it against the current tab's URL.

## Changes Made

### 1. extension/popup/popup.js — URL guard on cached result restore (line 47)
- Changed `if (lastResult)` → `if (lastResult && lastResult.url === currentUrl)`
- Ensures cached results only display when they belong to the current tab URL

### 2. extension/popup/popup.js — URL guard in `pollForResult()` (line 183)
- Added early-return when `!result || result.url !== currentUrl`
- Stops polling gracefully if user navigated away from the URL that started the query
- Split the original `!result || status !== 'running'` check into two: URL mismatch and status completion

### No other files changed
- background.js already stores `url` on all result states (running/complete/error)
- result/result.js intentionally shows whatever was popped out
- tests/background.test.js tests storage logic which is correct
