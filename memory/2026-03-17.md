# 2026-03-17 (session 5) ‚Äî Fix sendMessage: Loader and "Ask Claude" Not Working

## Problem
Safari MV3's `browser.runtime.sendMessage()` returns a Promise ‚Äî the second argument is NOT a callback (that's the `chrome.*` pattern). The old code `browser.runtime.sendMessage(message, resolve)` caused Safari to interpret `resolve` as options/extensionId, discard the returned Promise, and leave the outer `new Promise` never resolving. This blocked `DOMContentLoaded`'s `Promise.all`, keeping the button disabled.

## Changes Made

### 1. sendMessage fix in all three extension files
- `extension/popup/popup.js`, `extension/settings/settings.js`, `extension/result/result.js`
- Replaced callback-based `new Promise((resolve) => browser.runtime.sendMessage(msg, resolve))` with `async function sendMessage(msg) { return await browser.runtime.sendMessage(msg); }` + try/catch returning null on failure

### 2. popup.js ‚Äî DOMContentLoaded try/catch
- Wrapped initialization `Promise.all` in try/catch to surface errors instead of hanging silently

### 3. popup.js ‚Äî Null/unknown status handling in click handler
- Added `else` branch after `error`/`complete` checks to re-enable button and show error on null or unexpected status

### 4. settings.js ‚Äî Null result check on save
- Added `!result` guard before success message so a failed save doesn't falsely confirm

### 5. background.js ‚Äî .catch() on handler chain
- Added `.catch()` to `handler().then(sendResponse)` for defense-in-depth if port is closed

### 6. tests/popup.test.js ‚Äî Updated mocks + 3 new tests
- Changed mock `sendMessage` from callback pattern `(msg, callback) => callback(...)` to Promise-based `(msg) => Promise.resolve(...)`
- Updated 2 poll test overrides to match
- Added Test 8: null result from click handler shows error
- Added Test 9: unknown status from click handler shows error
- Added Test 10: DOMContentLoaded error is caught and displayed

# 2026-03-17 (session 6) ‚Äî Fix Storage Quota Exceeded Error: Native Disk Storage

## Problem
Safari MV3 extensions have a ~5MB `browser.storage.local` quota. Storing full Claude CLI responses (10-100KB+ each) as `lastResult` caused `"Exceeded storage quota"` errors.

## Solution
Move `lastResult` storage to native disk via the existing Swift `SafariWebExtensionHandler` bridge. Keep only small data (`settings`, `history`) in `browser.storage.local`.

## Changes Made

### 1. SafariWebExtensionHandler.swift ‚Äî 3 new actions + helper
- `storageDirectory()` ‚Üí returns `<container>/Library/Application Support/ClaudeAssistant/`, creates if needed
- `handleStoreResult` ‚Üí serializes result dict to JSON, writes to `lastResult.json` atomically
- `handleGetStoredResult` ‚Üí reads `lastResult.json`, returns `{ result: <data> }` or `{ result: null }`
- `handleClearStoredResult` ‚Üí deletes `lastResult.json` if exists, always returns success

### 2. background.js ‚Äî Rewired storage, reduced limits, migration
- `saveLastResult()` ‚Üí `sendNativeMessage({ action: 'storeResult', result })` with try/catch
- `getLastResult()` ‚Üí `sendNativeMessage({ action: 'getStoredResult' })` with try/catch, fallback null
- `clearLastResult` message handler ‚Üí `sendNativeMessage({ action: 'clearStoredResult' })`
- History cap: 50 ‚Üí 25 entries
- History preview: 200 ‚Üí 100 chars
- `onInstalled`: added `browser.storage.local.remove('lastResult')` migration
- Exported `sendNativeMessage` for test access

### 3. tests/background.test.js ‚Äî Updated mocks + new tests (33 total)
- Mock `sendNativeMessage` routes `storeResult`/`getStoredResult`/`clearStoredResult` to in-memory object
- Added `browser.storage.local.remove` mock
- Updated history cap test: 50 ‚Üí 25
- New: `saveLastResult` sends `storeResult` action with correct payload
- New: `getLastResult` sends `getStoredResult` action
- New: `saveLastResult` does not throw on native message failure
- New: `getLastResult` returns null on native message failure

### 4. ARCHITECTURE.md ‚Äî Updated
- Component diagram shows split storage (browser.storage.local + native disk)
- Storage schema section split into browser storage and native disk schemas
- Design decisions updated: `lastResult` on native disk rationale, history cap rationale

# 2026-03-17 (session 8) ‚Äî Fix Pop-out Empty + Cached Results Not Showing

## Problem
Two bugs: (1) pop-out page (result.html) was empty even when popup just displayed content, (2) cached result not restoring when reopening popup for a previously-queried URL. Root cause: `background.js` message listener used callback-based `sendResponse` + `return true` pattern, which is unreliable in Safari MV3 ‚Äî the response can be lost before `sendResponse` fires when the popup closes or the service worker cycles.

## Changes Made

- Safari MV3 natively resolves the returned Promise as the response ‚Äî no `sendResponse` callback, no `return true`, no channel-closing race conditions

# 2026-03-17 (session 10) ‚Äî Per-URL Result Caching

## Problem
The extension used a singleton `lastResult.json` file. When a new URL was processed, the previous URL's result was overwritten. Going back to a previously-processed URL found no matching cache.

## Solution
Store per-URL results in a `results/` subdirectory using SHA-256-hashed filenames. Keep `lastResult.json` as a "most recent" pointer for the pop-out view. Add LRU eviction (25 files max).

## Changes Made

### 1. SafariWebExtensionHandler.swift ‚Äî Per-URL storage + eviction
- Added `import CryptoKit`
- `resultFileURL(for:)` ‚Äî SHA-256 hashes URL, takes first 16 hex chars, returns `results/<hash>.json`
- `evictOldResults()` ‚Äî sorts `results/` by mtime, deletes files beyond position 25
- `handleStoreResult` ‚Äî now also writes to per-URL cache file when result has a `url` field
- `handleGetStoredResult(message:)` ‚Äî accepts URL param, looks up per-URL cache first, falls through to `lastResult.json`
- `handleClearStoredResult(message:)` ‚Äî accepts URL param, clears that URL's cache file; without URL clears `lastResult.json`

### 2. background.js ‚Äî URL-aware getLastResult
- `getLastResult(url)` ‚Äî accepts optional URL, includes in native payload
- Message handler `getLastResult` case ‚Äî forwards `message.url`
- Message handler `clearLastResult` case ‚Äî forwards `message.url` to native handler

### 3. popup.js ‚Äî Pass URL to fetch
- `DOMContentLoaded` ‚Äî removed `getLastResult` from `Promise.all`, fetches after determining `currentUrl`
- `pollForResult` ‚Äî passes URL in poll requests

### 4. result.js ‚Äî No changes
Still calls `getLastResult` without a URL, falls through to `lastResult.json`

### 5. Tests
- background.test.js: Updated mock with `perUrl` map, added per-URL payload tests, isolation test, clear test, handler forwarding test (71 total)
- popup.test.js: Track `getLastResultMessages`, verify URL included, updated Test 2 for per-URL miss (30 total)
